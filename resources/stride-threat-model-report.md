# STRIDE Threat Model Report for DataBridge Pro

## Introduction: goals, scope, and required inputs

This STRIDE analysis focuses on DataBridge Pro's guided MySQL migration workflow, including the React wizard, the Vite `/api/*` middleware, and source-to-target MySQL data flows documented in the architecture. The goals are to identify threats that could affect database credentials, schema metadata, mapping plans, migrated data, and audit readiness; validate those threats against controls already present in the implementation; and provide mitigation and verification actions that teams can use during design reviews and audits. The current scope covers local/development-server and preview-server usage of the migration assistant, browser-to-middleware POST requests, middleware-to-MySQL connections, table/schema discovery, queued mapping plans, and mapped batch transfers. Inputs reviewed were `docs/architecture.md`, `README.md`, and `testDbConnectionPlugin.ts`; the referenced `resources/implementation-playbook.md` was not present in the repository, so this report records that missing input as a constraint and should be revisited when the playbook becomes available.

## STRIDE threat analysis and control validation

**Spoofing** risks include unauthorized users invoking `/api/test-db-connection`, `/api/list-tables`, `/api/table-schema`, `/api/table-foreign-keys`, or `/api/mapped-transfer` because the middleware does not authenticate callers before accepting connection details or transfer jobs. **Tampering** risks include malicious mapping plans or table/column identifiers altering unintended data; existing identifier allow-listing and backtick quoting reduce SQL injection risk, and parameterized values reduce data-value injection risk, but there is no server-side authorization model to prove a requested source, destination, table, or column is permitted for that operator. **Repudiation** risks include operators denying who tested a connection, enumerated schemas, changed mappings, or executed transfers because current responses provide immediate status but no durable user identity, request ID, immutable audit log, or signed transfer record. **Information Disclosure** risks include exposure of database credentials in browser-to-server POST bodies, schema metadata returned to any caller who can reach the dev server, and raw MySQL error messages returned to clients; the current server-side database execution avoids direct browser-to-database connectivity, but credentials are still transmitted to the middleware and errors may reveal host, schema, table, or constraint details. **Denial of Service** risks include large JSON bodies, repeated schema enumeration, expensive `LIMIT/OFFSET` transfers, high batch counts, or long-running database connections; the implementation caps batch size at 2,000 and closes MySQL connections in `finally`, but request body size, request rate, timeout, and concurrency limits are not yet enforced. **Elevation of Privilege** risks include using overly privileged MySQL accounts from the UI to read or write beyond the intended migration scope, plus exposing the same middleware during preview where it can still perform database operations; current host/database presence validation and identifier constraints do not enforce least privilege, role separation, or environment gating.

## Actionable mitigations

Add an explicit access-control boundary before all `/api/*` handlers, such as session authentication, CSRF protection for browser-originated requests, and an authorization policy that maps users to allowed projects, databases, schemas, tables, and operations. Treat all connection inputs and migration plans as sensitive: require TLS to the app and MySQL where supported, avoid persisting secrets in frontend state longer than needed, redact credentials and sensitive MySQL details from UI logs and API errors, and replace raw exception messages with safe error codes plus server-side diagnostics. Strengthen integrity controls by validating that requested source/target tables and columns exist immediately before execution, comparing mappings against fetched schemas, requiring explicit confirmation for destructive or high-volume jobs, and using least-privilege MySQL accounts with read-only permissions on sources and narrowly scoped insert/update permissions on targets. Add auditability by recording authenticated actor, timestamp, source/destination identifiers, selected tables, mapping checksum, transfer job ID, row counts, duplicate-key behavior, and success/failure outcome in an append-only audit sink. Improve resilience with JSON body limits, request timeouts, rate limits, maximum job counts, safe concurrency limits, transfer cancellation, progress checkpoints, and database-side statement timeouts.

## Verification and training examples

Verification should include automated and manual checks for each mitigation: send unauthenticated requests to every `/api/*` route and confirm they are rejected; attempt unauthorized table and column mappings and confirm authorization denies them; submit invalid identifiers and confirm the existing identifier validation still blocks them; trigger duplicate-key and connection failures and confirm responses are redacted while full diagnostics are available only in protected server logs; run a large transfer test and verify body-size, batch-size, timeout, rate-limit, and connection-cleanup behavior; and inspect the audit log to confirm every test connection, schema read, and transfer execution has a non-repudiable actor and job ID. As a training example, an online-banking-style migration would emphasize spoofing controls such as multi-factor authentication and per-account authorization before transfer execution, while a healthcare-style migration would emphasize information-disclosure controls such as encryption in transit, least-privilege data access, and audit logging for patient-data schema reads. Overall, DataBridge Pro already contains useful baseline controls for server-side execution, parameterized queries, identifier quoting, batch processing, and connection cleanup, but audit readiness depends on adding identity, authorization, redaction, resource governance, and durable evidence that recommended controls are operating effectively.
